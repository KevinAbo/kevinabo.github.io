    # 前言

    ​ 本文档描述媒体方 Media 通过客户端或者服务端方式请求 AdExchange 系统(文 档后用 AdExchange 简称)的通用接口。

    ​ 基本流程如下：

    1. Media 向 AdExchange 发送广告请求(BidRequest)，具体数据格式参看章节1

    2. AdExchange 内部和下游的 DSP 等平台进行 RTB 通信，通过竞价产生出合适的广告，

      得到本次投放的广告素材

    3. AdExchange 向 Media 返回广告响应(BidResponse)，具体数据格式参看章节2

    # 接口说明

    所有参数均采用 UTF-8 编码。接口为POST接口

    POST /bid path HTTP/1.1
HOST: dsp host
Content-Type: application/json
Content-Length: length
Connection: Keep-Alive

    # 对接测试流程

    ​ 在商务上完成协商后，双⽅方开放⼈人员建⽴立联系，完成信息的同步后进⾏行行开发，提交相关配置信息;

    ​ 确定基本没问题后，进⾏行行沙箱环境自测。自测结束需要联系AdxExchange运营及时确认AdxExchange成功响应和数据上报链路。

    ​ 完成测试后上线进⾏行行指定媒体的测试放量量，通常为5-10CPM，投放完成后根据双⽅方的报表系统延迟协商核对数据的时间，拉取双⽅方数据，与第三⽅方监控数据，确定没有异常gap后进⾏行行商务上的正式投放。

    # 1. Bid Request说明

    媒体 向AdxExchange 发送竞价请求，竞价信息保存在BidRequest对象中，具体说明如下：

    # Bid Response

    一个 BidResponse 中可包含多个 SeatBid 对象。一个 SeatBid 代表DSP端的一个广告主。 SeatBid 中又可包含多个 Bid 对象，一个 Bid 代表一个参与竞价的广告。 正常出价返回 HTTP 200 OK，不出价返回 204 no content，100ms内，无响应的，视为竞价异常。

    字段 类型 必须 说明
    id string Y response id 和request id保持一致
    seatbid object[] N 广告主列表，如果参与竞价，必须包含一个seatbid对象
    bidid string N 由DSP生成的response id，用于 logging/tracking

    SeatBid Object

    字段 类型 必须 说明
    seat string Y adx为广告主分配的广告主id。
    bid object[] Y 竞价信息

    Bid Object

    字段 类型 必须 说明
    id string Y dsp生成的bid id，用于追踪或打log
    impid string Y bid request中对应的Imp对象id
    price float Y 出价，单位：分
    dealid string N PDB、PD模式必选
    crid string Y 创意id，需素材审核的广告位该字段不为空
    nurl string N 竞价成功通知url，需要支持宏替换，见章节3.2
    adm object N 广告物料，Native对象，非素材审核的广告位该字段不为空
    imp_trackers string[] N 曝光上报地址，需要支持宏替换，见章节3.2
    click_trackers string[] N 点击上报地址，需要支持宏替换，见章节3.2
    down_start_trackers string[] N 下载开始上报地址，需要支持宏替换，见章节3.2
    down_comp_trackers string[] N 下载完成上报地址，需要支持宏替换，见章节3.2
    install_trackers string[] N 安装成功上报地址，需要支持宏替换，见章节3.2
    video_trackers object[] N 视频监测链接数组，需要支持宏替换，见章节3.2
    dp_trackers string[] N dp唤起成功监测链接，需要支持宏替换，见章节3.2

    adm 非必填，系统根据审核过的创意ID填充广告物料。

    Video Tracker Object

    # 3. 监测上报和宏替换说明

    # 3.1 上报说明

    1. 媒体对竞价成功的 DSP，发送获胜通知（nurl），协议为HTTP GET；
    2. 在广告被曝光时并行上报多组展现监测(imp_trackers)
    3. 在广告被点击时，并行上报多组点击监测（clk_trackers），同时进行跳转
    4. 对于有下载监测需求的广告，需要回传下载进度
    5. 对于有deeplink唤起监测需求的广告，需要回传deeplink唤起数

    # 3.2 宏替换说明

    需要支持以下宏替换：

    Macro Descrption
    ${AUCTION_ID} BidRequest.id
    ${AUCTION_CRID} BidResponse.bidseat.bid.crid
    ${AUCTION_PRICE} 竞拍价(次高价结算)，rtb结算下需要替换imp_trackers中该宏，单位为分
    ${AUCTION_DX} 鼠标或手指按下时相对于素材的X坐标，clk_tracking/dplink/land需要替换
    ${AUCTION_DY} 鼠标或手指按下时相对于素材的Y坐标，clk_tracking/dplink/land需要替换
    ${AUCTION_UX} 鼠标或手指弹起时相对于素材的X坐标，clk_tracking/dplink/land需要替换
    ${AUCTION_UY} 鼠标或手指弹起时相对于素材的Y坐标，clk_tracking/dplink/land需要替换

    # 6.1 Deeplink集成

    • deeplink 在发展过程中衍⽣生了了ShemeLink 和 UniversalLink。本系统中dplink 指schemelink 地 址;
    • 当返回dplink时，ctype需要返回3，同时落地⻚页landing字段返回h5点击地址，作为 deeplink唤起失败的打底地址;
    • 若投放为universal link 地址;universal link 是作为落地⻚页放bidResponse.adm.land字段 中,此时ctype需要返回3;
